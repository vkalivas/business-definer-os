# Business Definer OS — Operating Context

## What This Repository Is

A personal business-definition wiki and MVP readiness system.

Raw material comes from Claude chats as HTML and markdown artifacts.
The repo collects, organizes, and analyzes those artifacts to maintain a live,
honest picture of what is defined, what is weak, and what to work on next.

## Two Classes of Files

- **Source artifacts** — raw HTML, markdown, and other outputs from Claude chats.
  Preserved as-is. Never generated by the control system.
  Live in `artifacts/raw/` (unfiltered) and `artifacts/curated/` (active working library).

- **Derived control files** — indexes, maps, readiness views, gap trackers, summaries,
  and next-work recommendations. Live in `control/`.
  These are how the system thinks about the source artifacts.

Never conflate the two.

## Artifact Areas

- `artifacts/raw/html/` — all HTML outputs from Claude chats, unfiltered
- `artifacts/raw/markdown/` — all markdown outputs from Claude chats, unfiltered
- `artifacts/curated/html/` — HTML artifacts still considered useful and active
- `artifacts/curated/markdown/` — markdown artifacts still considered useful and active

## Control Files

- `control/index.md` — what exists, one-line summaries, layer mapping
- `control/layer-readiness.md` — readiness status across all 8 MVP definition layers
- `control/gap-tracker.md` — known gaps, weak areas, explicit assumptions, open questions
- `control/next-work.md` — current opinionated recommendation for what to work on next
- `control/current-beliefs.md` — current best view: product thesis, MVP shape, key assumptions, unresolved tensions
- `control/rabbit-holes.md` — interesting but non-blocking ideas that should not drive MVP decisions yet

## The 8 MVP Definition Layers

| # | Layer | What it covers |
|---|-------|----------------|
| 1 | Problem / Opportunity | Why this product should exist |
| 2 | User / Customer | Who it's for and why they'd care |
| 3 | Product Concept | What it is and what job it does |
| 4 | Core Workflow / User Flow | How the user moves through it |
| 5 | Output / Deliverable Structure | What it produces and what "good" looks like |
| 6 | System Shape / Core Objects | Main entities and structural components |
| 7 | MVP Boundary | What's in, what's out, minimum coherent version |
| 8 | Validation / Open Questions | What's assumption-based, what most needs testing |

## Readiness Levels

- **Not Started** — not meaningfully defined
- **Mentioned** — referenced but not thought through
- **Considered** — assumptions exist, still loose
- **Structured** — clearly defined enough to guide related work
- **MVP-Ready** — good enough for design/build to proceed

A layer is MVP-Ready when: assumptions are explicit, logic is coherent, downstream work can proceed responsibly,
and remaining risk is acceptable for MVP.
Full research or validation is not required.

## Prioritization Labels

- **Required Now** — blocking MVP progress today
- **Required Soon** — will block MVP progress if not addressed
- **Helpful but Not Blocking** — adds value but doesn't gate anything
- **Rabbit Hole / Non-Blocking** — interesting but not on the critical path
- **Blocked by Another Artifact** — can't progress until a dependency resolves

## Blockers vs Rabbit Holes

An MVP blocker gates downstream work. Without it, building would be incoherent or reckless.
A rabbit hole is interesting but doesn't gate anything — a reasonable assumption can stand
in its place until later. If something doesn't block downstream work, it's a rabbit hole
until proven otherwise.

## Operating Style

- Opinionated — makes specific calls on what matters most
- Honest about uncertainty — marks gaps, assumptions, and open questions clearly
- Practical over formal — optimizes for readability and usability
- Sequencing-aware — thinks in dependencies and critical paths, not just checklists
- Lean — avoids redundant files and premature formalization
- Direct — recommendations say what to do and why, not just what's missing

## What "Next Work" Can Mean

The right next move is not always creating a new document. It may be:
- creating a missing artifact
- merging overlapping artifacts
- synthesizing partial artifacts into one stronger one
- creating a control map or readiness view
- making assumptions explicit so downstream work can proceed

## Default Operating Protocol

When new artifacts are added, follow this sequence:

1. Inspect the new source artifacts without rewriting them.
2. Summarize what each artifact contributes.
3. Map each artifact to one or more of the 8 MVP definition layers.
4. Note whether each artifact:
   - strengthens an existing layer
   - duplicates another artifact
   - conflicts with another artifact
   - exposes a new gap
   - belongs in a rabbit hole
5. Update the control files as needed:
   - control/index.md
   - control/layer-readiness.md
   - control/gap-tracker.md
   - control/current-beliefs.md
   - control/next-work.md
   - control/rabbit-holes.md
6. Recommend the single highest-leverage next move.

Do not create new polished business documents unless asked.
Prefer understanding, mapping, synthesis, and prioritization first.

## Current Focus

The current goal is not to fully formalize the business.
The current goal is to build and maintain a personal business-definition wiki that:
- preserves useful Claude-generated artifacts
- makes them browsable and understandable
- shows what has already been defined
- shows what is weak or missing
- recommends what should be worked on next to reach a coherent MVP
